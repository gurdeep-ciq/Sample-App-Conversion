UI Navigation Walkthrough · companion to architecture-v2 (with v3 deltas noted)

BrewGenie — Screen-by-Screen

Nine static mockups showing how the operator moves from "blank slate" to "live pipeline" — and exactly where the business rules live.

⚙️ v3 UI deltas (from May 11 review with Shyam — see brewgenie-architecture-v3.html)

The screens below are v2 mockups. After the May 11 review, the following UI changes are locked in for v3. Screens will be re-rendered in a follow-up doc.

  • Screen 1 (Dashboard): Remove the + New Customer button. Customer creation lives in BevGenie / ConsumerIQ only. BrewGenie only searches/lists existing customers.
  • Wizard breadcrumb: 6 steps → 7 steps. New order: Upload · Classify · Template · Configure · Master · Dry-run · Publish. Classify (AI) is the new step 2; Master entities (mastering service) is the new step 5.
  • New screen between Upload and current "Profile": Classification — Claude returns the data category (1 of 14) plus confidence; operator confirms or overrides from a dropdown.
  • New screen between Configure and Dry-run: Mastering Review Queue — operator confirms low-confidence entity matches (brand, account, person, distributor) or creates new global golden records.
  • Screen 6 (Rules): reframe from "Sera Luce schema" to "Component templates". The five rule tabs (Universal / Filter / Transform / Alias / Custom) survive, but they're now scoped to the component template for the standardize step, not the whole pipeline.
  • Screen 9 (Pipeline Monitor): Remove Open in Dagster ↗ button. BrewGenie's UI is Dagster (forked open-source UI with our skin). Add inline DQ summary widget showing per-step null rates and anomaly counts.
  • New Sidebar item: Template Library — browse + edit specific templates (per-vendor-per-category) and the underlying component templates.
  • Vocabulary: "distributor" → "data vendor" wherever it's not literally about a beer-wholesaling distributor (the table, the search filter, the breadcrumb).
  • Env labels: "Amber Tree" → "Staging (Amber Tree)" · "Supabase prod" → "Production (Supabase)" wherever environment labels appear.

Below is the v2 walkthrough preserved as-is for reference — it still shows the operator/customer view dichotomy and the rule-editor design intent, both of which carry over to v3.

Operator view

For Twenty20 staff onboarding new customers

Full chrome — every control visible, including confidence scores, fact/dim tags, dry-run row counts, audit trail links. Used during the white-glove onboarding phase (first 5-10 customers).

Customer view

For non-technical founders running their own pipelines

Same six steps, simplified language. Rules become plain-English toggles. Schema mapping is hidden unless something needs human judgment. Goal: a founder like Justin can self-serve in < 30 min.

Same routes, same backend, same publish action. The difference is a feature flag at the user level (user.role = 'customer' | 'operator') that swaps chrome and copy. Each screen below shows the operator view with a teal callout describing what changes for the customer.

Notable elements

  • Fact vs. Dimension tags — visual cue for star-schema modeling. Operator doesn't have to know SQL; the tags say what each field is for.
  • Curved connectors rendered with React Flow (@xyflow/react). Operator drags from source pill to target pill to map. Existing connections (AI's suggestion) can be unlinked.

Customer view — mostly hidden

  • This screen usually doesn't render at all for customers. If the AI mapped every column confidently, BrewGenie auto-advances from Screen 4 → Screen 6, skipping mapping entirely.
  • When it does surface: only the unmapped/ambiguous columns appear, framed as questions. Example: "We weren't sure what 'RepCode' means in your file — is this the salesperson's ID? Or something else?"
  • No fact/dim language. Tags hidden. The whole screen presents as "Help us understand this column" with a dropdown of plain-English options.
  • If everything is mapped: a one-line summary appears in the next screen instead: "We matched all 8 columns in your file."

The five rule categories — what each one captures

① Universal (applied to every distributor)
  • Zero-quantity row drop
  • Out-of-state row drop (per distributor territory)
  • State name typo correction DCB → DC
  • Division casing unification OFF Premise → Off Premise
  • Customer name alias merge
  • Sales rep code variant merge
② Filter rules (drop rows matching criteria)
  • Artisans: drop MoCo DLC placeholder rows
  • Artisans: drop DLC-* store-level rows
  • Vintage: drop where Division ∈ {SalesRep, Employee, Supplier}
  • Vintage: drop customer-info-only rows (no Dollars Sold)
  • Curated: drop Account Type = Samples
  • Curated: drop Account = Employee_*
  • Curated: drop Division = NA
  • Curated: drop Price Label blank + FOB blank/zero
  • PAL: drop where Price Point Name ∌ "Sera Luce"
  • PAL: drop manual subtotal rows (no Date)
  • Banville: drop Division = NA
  • Orsini: drop Division ∈ {SalesRep, Salesrep, Sales Rep}
③ Transform rules (conversions & derivations)
  • PAL 4-bucket classifier: Item × Price Point Name → bucket
    • Wholesale keg → qty × 1
    • Wholesale case → qty × 1
    • Retail 4-pack → qty ÷ 6
    • Retail single can → qty ÷ 24
  • Orsini volume discount: if qty ≥ 5, subtract $2.50/unit
  • Vintage: net revenue = Dollars Sold (post-RIPs)
  • Banville: per-row variable unit_price
  • Curated hybrid pricing: explicit FOB Price when present, A&V wholesale fallback otherwise
④ Alias rules (entity reconciliation)
  • Customer aliases: free-form mapping of raw names → canonical (e.g. "THE ITALIAN STORE II" = "Italian Store II")
  • Sales rep aliases: code & name variants → single rep (e.g. JGARERI / JOE → Joe Gareri)
  • State code normalization: typos, casing
  • Distributor canonicalization: link to NBWA registry via nbwa_distributor_id
⑤ Custom Python (engineering escape hatch — flagged for review)
  • Marked "needs custom code" — visible in audit log; engineer adds a transform function and re-registers the template before publish can proceed.
  • Used only for logic that's genuinely beyond declarative rules (e.g. cross-table joins, calls to external services, ML lookups).
  • Today's Sera Luce pipeline has zero rules in this category — all 6 distributors codify cleanly with categories ①–④.

Customer view — fundamentally different. Full mockup below.

  • No tabs. Tabs (Universal / Filter / Transform / Alias / Custom) are operator concepts. Customer just sees one input: "Anything we should ignore from this file?"
  • No structured JSON. The rule-card-body block (field: Division · op: in · value: ["SalesRep"...]) is hidden behind a "Show as code" toggle.
  • Toggles, not Add/Edit/Delete. AI-suggested rules show up as cards with simple on/off switches. Off = the rule won't apply. No drag-handles, no rule-ordering.
  • Plain-English chat input is the primary action. Type "drop employee sales" → Haiku compiles → a new toggle card appears with the rule pre-applied.
  • Universal rules are entirely hidden — they apply automatically to everyone and are noise to a customer.
"I want to type 'skip employee sales' and have BrewGenie just do it. The tabs and JSON would lose me in 10 seconds."

What's different about the customer rules screen

  • One input, not tabs. The plain-English chat input is the primary interaction. The structured tabs (Universal / Filter / Transform / Alias / Custom) are operator-only — they exist server-side but never render for customers.
  • Toggles instead of buttons. AI-suggested rules ship pre-toggled-on for high-confidence detections, pre-toggled-off for low. Customer flips a switch to dis-/en-able. No "edit" / "delete" — only on/off.
  • "Skip" verb everywhere. Replaces "drop", "filter", "exclude" — softer, less destructive-sounding, still precise.
  • Impact in human units. "drops ≈12 rows" not "drops 12 rows out of 4,127 with 0.29% volume impact". Approximate ≈ signals "we estimated this from your sample".
  • "Show as code (advanced)" toggle at the bottom expands to the operator structured-form view. For curious customers (or support sessions), nothing is hidden — just out of the way.
  • Universal rules hidden. Zero-qty, state typo, division-casing rules apply to everyone and aren't worth showing.

  • Stat cards are live — they refresh every 30 seconds via Dagster's GraphQL API.
  • "Open in Dagster" button is the escape hatch: anything Dagster's UI can do that BrewGenie can't (e.g. inspect asset materializations, view raw logs) is one click away.
  • No write controls on this screen. BrewGenie deliberately doesn't let operators trigger runs from here — that should go through the Pipelines section's "Replay" flow which has its own confirmation.

Customer view — "Data Health"

  • Renamed to "Data Health". The word pipeline doesn't appear anywhere on the customer's version of this screen.
  • Big traffic-light indicator dominates the page — green / yellow / red instead of run IDs and durations. Green = files arriving on schedule and processing cleanly.
  • Stat cards reduce to two: "Last update" (e.g. "2 hours ago") and "Files received this month" (e.g. "21"). Run counts, durations, and replay options are operator-only.
  • "Open in Dagster" button hidden. The escape hatch to the Dagster UI is operator-only.
  • Recent runs table replaced with a friendly "Recent files" list: filename · received-at · status. No run IDs, no row-in/row-out counts.
  • "Something looks wrong?" support link at the bottom — clicks open a pre-filled help request that includes the source ID, last run ID, and customer ID. Operator triages from BrewGenie's admin view.

Implementation note

  • One codebase, one route tree. Every page renders both views; a single React component-level prop (persona: 'operator' | 'customer') decides which copy, controls, and badges to show.
  • Auth determines persona. Twenty20 employees get user.role = 'operator' at sign-in; everyone else gets 'customer'. Role check happens server-side in middleware; client just renders accordingly.
  • Operators can preview customer view. A small toggle in the operator's avatar menu — "View as customer" — re-renders the current screen as it would appear to that customer. Useful during onboarding screen-shares and QA.
  • The customer mode is the bar. If a screen is intuitive in customer mode, it's certainly intuitive in operator mode. We design customer-first and let operators see more, not the other way around.
